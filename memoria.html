Introducción

Esta memoria documenta el desarrollo de la Práctica 1 de la asignatura Modelado y Tecnologías de Integración de Sistemas (MTIS), cuyo objetivo es implementar un sistema distribuido de gestión de edificio inteligente haciendo uso de dos paradigmas de servicios web: SOAP y REST.

El dominio del problema se centra en la gestión de empleados, el control de acceso a salas y el seguimiento de presencia dentro de un edificio, integrando además un servicio de notificaciones por correo electrónico.

La práctica ha supuesto un primer contacto directo con la creación de servicios web interoperables desde cero: definición de contratos (WSDL y OpenAPI), implementación de la lógica de negocio y despliegue de múltiples servicios que comparten una misma base de datos relacional.

Arquitectura del Sistema

El sistema está compuesto por ocho microservicios independientes, divididos en dos grupos según el protocolo que implementan.

Servicios SOAP

Los servicios SOAP siguen un enfoque contract-first: primero se define el contrato WSDL y a partir de él se genera el código Java mediante el plugin de Apache CXF.

Infraestructura de soporte

Además de los servicios, el proyecto incluye los siguientes elementos de apoyo:

  • Base de datos MySQL (edificio_inteligente): almacena todas las entidades del dominio y las claves de autenticación (wskeys).

Implementación de los Servicios SOAP

Enfoque contract-first y generación de código

Para los servicios SOAP se ha seguido el enfoque contract-first: el WSDL es el punto de partida, y el plugin cxf-codegen-plugin de Maven genera automáticamente las interfaces y clases de datos en tiempo de compilación. Esto garantiza que la implementación cumple exactamente el contrato publicado, sin riesgo de divergencia entre la definición del servicio y el código.

La estructura de cada servicio SOAP es la siguiente:

servicio-X/
├── src/main/resources/wsdl/      <-- Contrato WSDL
├── src/main/java/.../config/     <-- Configuración del endpoint CXF
├── src/main/java/.../impl/       <-- Lógica de negocio (implementa PortType)
├── src/main/java/.../model/      <-- Entidades JPA
└── src/main/java/.../repository/ <-- Repositorios Spring Data

Autenticación con WSKey

Todos los servicios —tanto SOAP como REST— requieren una clave de servicio (wskey) que se valida contra la tabla wskeys de la base de datos. Si la clave no existe o no es válida, el servicio devuelve un error de autenticación sin procesar la petición. De este modo se impide el acceso no autorizado sin necesidad de un mecanismo de autenticación externo.

La clave de pruebas utilizada en el entorno de desarrollo es PRACTICA1_KEY_2024, insertada directamente en la base de datos mediante el script schema.sql.

Patrón Holder en JAX-WS

Una de las particularidades de la implementación SOAP es el uso del patrón Holder<T> de JAX-WS para los parámetros de salida. Dado que Java no permite devolver múltiples valores de forma nativa, los parámetros anotados con @WebParam(mode = Mode.OUT) se pasan como Holder<T>, y el framework los mapea automáticamente al XML de respuesta SOAP. 

@Override
public void validarNIF(
    @WebParam(name = "wskey") String wskey,
    @WebParam(name = "nif")   String nif,
    @WebParam(name = "resultado", mode = Mode.OUT)
        Holder<Boolean> resultado,
    @WebParam(name = "mensaje", mode = Mode.OUT)
        Holder<String> mensaje) {

    if (!wskeyRepository.existsByClave(wskey)) {
        resultado.value = false;
        mensaje.value = "WSKey no válida";
        return;
    }
    // ... lógica de validación
}

Validaciones de documentos españoles

El servicio servicio-validaciones implementa algoritmos de verificación para cuatro tipos de documentos:

  • NIF: 8 dígitos + 1 letra verificadora, calculada como el módulo 23 del número sobre la tabla TRWAGMYFPDXBNJZSQVHLCKE.
  • NIE: letra inicial (X/Y/Z) + 7 dígitos + letra verificadora. Se sustituye X→0, Y→1, Z→2 y se aplica el mismo algoritmo que el NIF.
  • NAF: 12 dígitos; los dos últimos son el resultado de calcular módulo 97 sobre los diez primeros dígitos.
  • IBAN: 24 caracteres para el formato español (ES + 22 dígitos); verificación conforme a ISO 13616 mediante reordenamiento y módulo 97.

Implementación de los Servicios REST

Diseño de recursos y especificación OpenAPI

Los servicios REST exponen recursos siguiendo las convenciones HTTP. Cada servicio cuenta con su especificación OpenAPI 3.0.3 en formato YAML, que documenta todos los endpoints, parámetros, cuerpos de petición y respuestas posibles antes de comenzar la implementación Java.

Respuestas estandarizadas

Todos los servicios REST devuelven un objeto JSON con los campos codigo y mensaje, además del recurso solicitado en las operaciones de consulta. Los códigos de estado HTTP usados son los estándar:

  • 200 OK — operación completada con éxito.
  • 201 Created — recurso creado.
  • 400 Bad Request — datos de entrada inválidos.
  • 401 Unauthorized — wskey ausente o inválida.
  • 404 Not Found — recurso no encontrado.
  • 409 Conflict — conflicto con un recurso existente (clave duplicada).

Servicio de notificaciones

El servicio servicio-notificaciones utiliza Spring Mail para enviar correos electrónicos en tres situaciones:

  1. Validación exitosa de un usuario — POST /api/notificaciones/usuario-valido/{nif}.
  2. Notificación de error — POST /api/notificaciones/error.
  3. Cambio de presencia en una sala — POST /api/notificaciones/presencia-sala.

En el entorno de desarrollo se emplea Mailpit como servidor SMTP local, lo que permite verificar los correos sin necesidad de un servidor de correo real. La interfaz web de Mailpit (puerto 8025) muestra los mensajes recibidos en tiempo real.

Pruebas

Pruebas manuales

Las pruebas manuales se han realizado con las siguientes herramientas:

  • SoapUI — servicios SOAP. Se construyen peticiones XML con la wskey correcta e incorrecta, verificando tanto el flujo nominal como los casos de error: NIF inválido, empleado inexistente, intento de duplicado, etc.
  • Postman — servicios REST. Se cubren todos los verbos HTTP y se verifica que los códigos de estado de respuesta corresponden a los definidos en la especificación OpenAPI.
  • Mailpit (interfaz web en http://localhost:8025) — para confirmar que los correos generados por el servicio de notificaciones llegan correctamente con el contenido esperado.

Cliente de pruebas automáticas

El módulo cliente/ contiene la clase ClienteMain.java, que ejercita todos los servicios de forma secuencial. Para los servicios SOAP utiliza JaxWsProxyFactoryBean de Apache CXF; para los REST, el cliente HTTP nativo de Java (java.net.http.HttpClient, disponible desde Java 11).

Dificultades Encontradas y Aprendizajes

Esta práctica ha sido mi primer desarrollo completo de servicios web interoperables, lo que ha implicado un proceso de aprendizaje progresivo en varias áreas.

SOAP y JAX-WS

El principal obstáculo con SOAP fue comprender el modelo de programación de JAX-WS, en particular el patrón Holder<T> para parámetros de salida. Inicialmente intenté devolver los valores de respuesta directamente como resultado del método, lo que no funciona cuando el WSDL define múltiples parámetros de salida. Una vez entendido que JAX-WS mapea los parámetros OUT mediante Holder, la implementación resultó coherente con el contrato.

También fue necesario comprender la diferencia entre el enfoque contract-first (WSDL → código) y el enfoque code-first (código → WSDL), y por qué el primero es preferible en entornos de interoperabilidad real: el contrato es independiente del lenguaje de implementación y puede ser consumido por clientes escritos en cualquier plataforma.

REST y OpenAPI

La parte REST fue más accesible gracias a la familiaridad con Spring MVC. La principal lección fue la importancia de diseñar bien el contrato OpenAPI antes de implementar: los cambios posteriores en los endpoints obligan a actualizar tanto el código como la especificación, lo que genera trabajo adicional que se puede evitar con un buen diseño previo.

Configuración del entorno distribuido

La gestión de ocho servicios independientes, cada uno con su propio puerto y configuración, exigió atención en el orden de arranque —la base de datos debe estar disponible antes que cualquier servicio— y en la consistencia de las credenciales en todos los ficheros application.properties.

Un error concreto fue configurar el puerto SMTP a 25 en lugar de 1025 (el que utiliza Mailpit por defecto). Detectarlo requirió revisar los logs de Spring Mail, lo que fue una buena práctica para aprender a diagnosticar problemas en sistemas distribuidos.

Validaciones de documentos

Implementar los algoritmos de validación de NIF, NIE, NAF e IBAN desde cero supuso consultar la normativa española correspondiente y contrastar los resultados con validadores externos para asegurar la corrección de los cálculos. Fue un ejercicio útil para entender que los servicios de validación deben ser rigurosos: un error en el algoritmo podría rechazar documentos válidos o aceptar documentos inválidos.

Reflexión general

La práctica ha permitido comprender en la práctica por qué la interoperabilidad es un problema relevante en sistemas distribuidos: cuando los servicios están implementados con tecnologías distintas, el contrato (WSDL u OpenAPI) es el único elemento que garantiza que un cliente puede comunicarse con un servidor sin conocer sus detalles internos.

Haber trabajado con ambos paradigmas en paralelo ha hecho más clara la diferencia en su filosofía de diseño: SOAP orientado a operaciones con tipado fuerte y contratos explícitos, REST orientado a recursos con un modelo más simple y flexible. Cada uno tiene su contexto de aplicación, y conocer ambos permite elegir el más adecuado según los requisitos del sistema.

Conclusiones

La Práctica 1 ha resultado en un sistema funcional compuesto por ocho servicios web que cubren las necesidades de gestión de un edificio inteligente. Los objetivos planteados —implementar servicios SOAP y REST, definir contratos previos a la implementación, compartir una base de datos relacional y añadir autenticación mediante wskey— se han cumplido en su totalidad.

Desde el punto de vista formativo, el desarrollo ha consolidado conceptos teóricos de la asignatura sobre interoperabilidad y ha introducido el uso de herramientas y frameworks habituales en el sector profesional: Apache CXF, Spring Boot, Spring Data JPA y OpenAPI. La experiencia de depurar un sistema distribuido con múltiples servicios también ha resultado formativa, ya que los errores pueden originarse en cualquier capa (red, configuración, lógica de negocio) y localizarlos requiere una metodología ordenada.

En definitiva, la práctica ha servido para trasladar los conceptos de interoperabilidad al plano práctico, con un sistema que, aunque está en un entorno local de desarrollo, refleja el tipo de arquitectura que se encuentra en sistemas reales de gestión empresarial.
